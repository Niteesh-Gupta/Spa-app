Step 1 — TM submits request
TM fills: Hospital/Dealer name (dropdown from DB), SKU(s), current price, requested price, SDR%, deal stage (Quotation / Negotiation / Closure), validity in days (max 365), and justification note.
Step 2 — System routes to correct approver
Based on SDR%, system auto-assigns to RSM / ZSM / NSM / CM. TM cannot choose approver. Request appears in approver's queue.
Step 3A — Approver APPROVES
Status → Approved. TM receives notification. A 14-day confirmation window opens. Clock starts from approval date.
Step 3B — Approver REJECTS
Approver must enter rejection reason. Status → Rejected. TM sees reason. TM can re-raise with one click — creates a new linked request (pre-filled), original stays as Rejected in DB forever.
Step 4A — TM confirms "Hospital Accepted" (within 14 days)
TM confirms hospital has accepted the rate. Status → Active. Validity clock starts from this confirmation date (not from approval date). Record stored in DB as an active deal.
Step 4B — TM confirms "Hospital Declined"
Hospital did not accept the approved rate. Status → Hospital Declined. Record stays in DB (audit trail). TM can re-raise with modified pricing — new linked request created.
Step 4C — TM does nothing for 14 days
System auto-lapses. Status → Lapsed. No deal happened. Record stays in DB. TM can still re-raise if needed.
Step 5 — Active deal: validity running
Status = Active. Validity_expires_at = confirmed_at + validity_days. TM dashboard shows days remaining. Alert sent when <14 days remaining.
Step 6 — Validity expires
System auto-sets status → Expired. TM receives alert. TM can re-raise with one click. New request created, linked to original via re_raised_from field.
04
Re-Raise Logic
When can TM re-raise?
Trigger
Request was Rejected by approver
Trigger
Approval Lapsed (no confirmation in 14 days)
Trigger
Hospital Declined the approved rate
Trigger
Deal Expired (validity period ended)
How re-raise works
1. TM clicks "Re-Raise" on original request
2. New request form opens — pre-filled with all original data
3. TM can modify: price, SDR%, validity, justification
4. New request gets a new SPA number
5. New request has re_raised_from = original_request_id
6. Original request remains in DB unchanged
7. New request goes through full approval flow again
⚠ Original records are NEVER deleted. Every request lives in the DB regardless of status. This is the audit trail. Re-raised requests are new records linked to the original.
